# Full App Audit & Security Report

## SECURITY ISSUES

### HIGH SEVERITY

| # | Issue | Location | Risk |
|---|-------|----------|------|
| 1 | **ProGuard/R8 disabled** | `app/build.gradle.kts:30` | `isMinifyEnabled = false` - APK can be reverse-engineered, exposing business logic, API calls, and Firebase config |
| 2 | **FCM token logged** | `SmartFinanceMessagingService.java:20` | `Log.d(TAG, "FCM Token refreshed: " + token)` - Token exposed in logcat, can be intercepted on rooted devices |
| 3 | **100 Firestore calls vs 50 error handlers** | Multiple files | Only 50% of Firestore operations have `addOnFailureListener` - silent failures on the other 50% |

### MEDIUM SEVERITY

| # | Issue | Location | Risk |
|---|-------|----------|------|
| 4 | **PIN stored in SharedPreferences** | `PinSetupActivity.java`, `PinHelper.java` | PIN stored as plain text in SharedPreferences - should use EncryptedSharedPreferences |
| 5 | **User role in SharedPreferences** | `ChooseRoleActivity.java:69`, `RegisterActivity.java:52` | Role stored locally - can be tampered on rooted device |
| 6 | **No input sanitization** | `ExpenseClaimAddActivity.java:104`, `LoanCompareActivity.java:128-130` | `Double.parseDouble()` without try-catch in some paths |
| 7 | **Exported launcher activities** | `AndroidManifest.xml:24,284` | `MainActivity` and `LoginFormActivity` are `exported=true` - expected for entry points but verify intent filters |

### LOW SEVERITY

| # | Issue | Location | Risk |
|---|-------|----------|------|
| 8 | **SharedPreferences "UserData"** | Multiple files | No encryption for local storage |
| 9 | **No certificate pinning** | Network layer | Firebase SDK handles TLS but no custom pinning for API calls |
| 10 | **No biometric auth option** | Auth flow | Only PIN/password - no fingerprint/face ID |

### Error Handling Gaps

- **50 Firestore calls missing `addOnFailureListener`** - If network fails, user gets no feedback
- Some `Double.parseDouble()` calls outside try-catch blocks
- `getIntent().getStringExtra()` calls without null checks in `UpdateBillActivity`

### Thread Safety
- `Handler` usage in `RegisterActivity.java` and `SavingDetailsActivity.java` - generally safe since they use main looper
